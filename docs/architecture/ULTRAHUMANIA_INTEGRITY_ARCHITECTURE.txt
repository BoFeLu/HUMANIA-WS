ULTRAHUMANIA INTEGRITY ARCHITECTURE
Updated: 2026-03-11

1. PURPOSE
This document defines the current integrity architecture baseline of ULTRAHUMANIA after trust-root separation, external signing, dual signing authority, signed baseline update workflow, and unified health-check validation.

2. CURRENT TRUST CHAIN
External signing authority
-> miniPC signer OR Pocophone/Termux signer
-> root_manifest.json.sig
-> C:\ULTRAHUMANIA_TRUST_ROOT\keys\allowed_signers
-> root_verify.ps1
-> secure_sentinel.ps1
-> C:\HUMANIA\sentinel\manifest.json
-> protected runtime files

3. ACTIVE TRUST-ROOT LOCATION
C:\ULTRAHUMANIA_TRUST_ROOT

- guardian.exe

- root_verify.ps1

5. REPOSITORY-SIDE OPERATIONAL TOOLS
- C:\HUMANIA\tools\Update-SignedRootBaseline.ps1
- C:\HUMANIA\tools\Get-UltrahumaniaSecurityHealth.ps1

6. SIGNING MODEL
The Windows host is verification-oriented.
Signing authority is externalized.
Validated external signing custodians:
- miniPC over Tailscale SSH
- Pocophone F8 Pro / Termux over SSH on port 8022

7. BASELINE UPDATE MODEL
A new root manifest is prepared in pending state.
The pending manifest is signed outside the main Windows host.
The signature is imported back into the trust root.
Verification and activation occur only after successful signature validation.
Post-activation checks:
- root_verify.ps1 must pass
- secure_sentinel.ps1 must pass

8. HEALTH CHECK MODEL
Unified command:
pwsh -NoProfile -ExecutionPolicy Bypass -File C:\HUMANIA\tools\Get-UltrahumaniaSecurityHealth.ps1

Expected final result:
OVERALL STATUS: OK

9. VALIDATED MILESTONES
- deterministic git hook runtime established
- generated INFRA_MAP removed from tracked baseline logic
- oversized Git object incident resolved
- signed trust-root baseline validated
- first successful signed baseline apply recorded

10. CURRENT SECURITY EFFECT
- the main Windows host no longer needs to be the primary signing authority
- baseline signing can be performed from more than one external custodian
- verification remains local and reproducible
- integrity checks are operationally testable in a single command

11. CURRENT LIMITATIONS
- trust root remains on the same Windows host
- main verification logic is still PowerShell-based
- evidence growth policy must remain controlled
- legacy Windows private signing material policy must still be finalized if any archived copy remains
